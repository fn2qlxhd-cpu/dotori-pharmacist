도토리 약사님 알림 — 8개 중복 발생 원인 분석 및 최종 조치

=== 발견된 문제 ===

1) GitHub 저장소에 firebase-messaging-sw.js가 별도 파일로 남아있었음
   - service-worker.js 안에 FCM 백그라운드 핸들러가 이미 통합되어 있는데,
     GitHub Actions 커밋 로그에 "Update firebase-messaging-sw.js"가 보여
     저장소에는 그 파일이 별도로 존재하는 상태였을 가능성이 높음.
   - 만약 그 파일이 같은 scope('./')에 등록될 수 있는 상태라면, 두 서비스
     워커가 활성 자리를 두고 경쟁하면서 푸시 라우팅이 불안정해질 수 있음.
   - 조치: 이 zip에는 해당 파일이 없음을 확인. GitHub 저장소에서도
     firebase-messaging-sw.js 파일을 반드시 삭제해야 함 (수동 조치 필요).

2) concurrency.cancel-in-progress: true 의 위험성
   - 짧은 시간 안에 워크플로우 트리거가 여러 번 들어오면, 진행 중인 실행이
     sendFCM 호출 도중 강제 종료될 수 있었음. claim(예약)과 실제 전송 사이의
     비원자적 구간에서 캔슬이 끼어들면 상태가 불안정해짐.
   - 조치: cancel-in-progress: false로 변경. 겹친 실행은 취소하지 않고
     끝날 때까지 대기 후 순차 실행되도록 함.

3) deviceKey(userAgent 기반 기기 추정) 로직의 근본적 결함
   - 같은 기기가 PWA 재설치 시 userAgent 패치 버전이 미세하게 달라지는
     문제를 막으려 했으나, 정규화를 어떻게 해도 (a) 같은 모델의 서로 다른
     두 대를 같은 기기로 잘못 합치거나 (b) 패치 버전 차이로 같은 기기를
     다른 기기로 잘못 나누는 문제를 동시에 해결할 수 없었음.
   - 실제 시뮬레이션 결과 정규화를 강하게 걸면 사용자의 진짜 두 아이폰이
     하나로 합쳐져 한쪽이 영원히 알림을 못 받는 더 심각한 버그가 발생함.
   - 조치: userAgent 기반 deviceKey 로직 전체 폐기. pickLatestDocsPerDevice
     를 pickLatestDocsPerToken으로 교체하여, 오직 FCM 토큰 값 자체로만
     중복을 제어함. 시뮬레이션으로 3가지 시나리오(진짜 3대 기기 / 같은
     토큰 중복 문서 / 토큰 없는 문서 혼재) 모두 정상 동작 확인.

4) 무효 토큰 처리 방식 보강
   - 기존: FCM이 토큰 무효를 알려주면 fcmToken 필드만 null로 바꿈 →
     죽은 문서가 Firestore에 계속 쌓임.
   - 변경: 무효 토큰 확인 시 해당 문서 자체를 삭제.

=== 남은 미해결 가능성 (계속 관찰 필요) ===

- 위 조치들은 코드 레벨에서 발견 가능한 모든 구조적 결함을 제거했지만,
  "GitHub Actions 워크플로우가 실제로 그 시각에 몇 번 트리거됐는지"는
  Actions 탭 로그로 직접 확인하지 못한 상태임. cron-job.org에 09:00
  job이 중복 등록되어 있을 가능성은 여전히 열려있으므로, cron-job.org
  Cronjobs 목록에서 같은 시각으로 설정된 job이 1개뿐인지 다시 확인 필요.
